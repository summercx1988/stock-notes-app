# 盯盘笔记系统 - Debug 与股票分析开发路线图（v1）

**版本：** v1.0  
**日期：** 2026-03-24  
**适用周期：** 接下来 6-8 周

---

## 一、目标共识

本阶段只做两件事：

1. 把盯盘笔记做成轻量、稳定、快速的日常工具。
2. 在不牺牲速度的前提下，增强股票分析与复盘能力。

明确不做：

1. 不在本阶段引入图数据库。
2. 不把 RAG 作为主线功能。
3. 不做复杂策略回测平台化。

---

## 二、技术策略（AI CTO 建议）

### 2.1 架构分层

保持三层结构，避免“录入、检索、分析”耦合：

1. 文档真相层：`Markdown + YAML`（唯一真实数据源）
2. 查询索引层：本地索引（优先 SQLite）用于快速筛选与聚合
3. 分析引擎层：复盘统计、错因识别、提醒策略

### 2.2 产品约束

1. 主录入流永远轻量（录音/输入/保存不能被图表和重计算阻塞）。
2. K 线与复盘在独立模块惰性加载。
3. 复盘只统计 `看盘预测` 类别事件。

### 2.3 性能 SLO（阶段目标）

1. 冷启动：`< 2s`
2. 单条笔记保存：`< 100ms`（不含网络 ASR 时间）
3. 左栏检索：`< 80ms`
4. 时间轴筛选：`< 150ms`（1k 事件量级）
5. 复盘计算：缓存命中 `< 500ms`，缓存未命中 `< 2s`

## 三、执行计划（3 个 Sprint）

## Sprint 1（第 1-2 周）：稳定性与 Debug 基线

### 目标

打通并固化关键链路：录入 -> 保存 -> 时间轴 -> 复盘入口。

### 任务

1. 建立端到端回归样例（20-30 条）。
2. 建立结构化错误日志（分段：转写、提炼、匹配、写入、复盘）。
3. 清理已知阻塞类问题（股票匹配兜底、类别保存一致性、时间排序一致性）。
4. 增加“用户可恢复”能力（保存失败可重试、核心操作失败提示明确）。

### 代码焦点

1. `src/renderer/components/RecordingControl.tsx`
2. `src/main/services/ai-processor.ts`
3. `src/main/services/notes.ts`
4. `src/main/services/VoiceTranscriberClient.ts`

### 验收标准

1. 关键链路回归通过率 `>= 99%`
2. 无“保存流程卡死”类阻塞问题
3. 失败可定位到具体阶段

---

## Sprint 2（第 3-4 周）：性能与轻量化

### 目标

保持 Markdown 真相层不变，同时把查询和统计性能拉起来。

### 任务

1. 引入本地索引（SQLite）并建立事件索引表。
2. 左栏、时间轴、筛选改为“优先查索引，按需回读文档”。
3. 增加增量索引同步机制（新增/修改/删除事件触发局部更新）。
4. 复盘结果增加短期缓存（按 `scope + stock + range + interval + rule` 维度）。

### 代码焦点

1. 新增 `src/main/services/index-store.ts`（建议）
2. `src/main/services/notes.ts`（事件变更回调）
3. `src/main/application/notes-app-service.ts`
4. `src/renderer/components/Sidebar.tsx`
5. `src/renderer/components/StockTimelineView.tsx`

### 验收标准

1. 达到性能 SLO（见上）
2. 主录入链路性能不受索引更新拖累
3. 业务行为与当前版本一致（无语义回归）

---

## Sprint 3（第 5-8 周）：分析增强与错误提醒

### 目标

从“统计胜率”升级到“发现错误模式并提醒”。

### 任务

1. 建立错因字典（初版）：
   - 追高
   - 逆势
   - 无确认入场
   - 风控失效
   - 情绪化交易
2. 复盘结果新增错因标注字段（支持人工修正）。
3. 新增“错误提醒面板”：
   - 最近 N 天高频错因
   - 连续犯错预警
   - 改进动作建议
4. 新增复盘筛选：
   - 类别（固定看盘预测）
   - 观点方向
   - 错因标签

### 代码焦点

1. `src/shared/types.ts`（错因模型）
2. `src/main/core/review-evaluator.ts`
3. `src/main/application/notes-app-service.ts`
4. `src/renderer/components/ReviewAnalysisView.tsx`
5. 新增 `src/renderer/components/MistakeInsightPanel.tsx`（建议）

### 验收标准

1. 可输出“错因分布 + 高频错误提醒”
2. 提醒策略可配置（N 天窗口、阈值）
3. 对现有复盘统计无破坏

---

## 四、工程治理建议

### 4.1 测试策略

1. 单元测试：`review` 算法、时间对齐、类别过滤
2. 集成测试：`notes` 的增删改查与排序重写
3. E2E 测试：录音上传、股票匹配、保存、时间轴筛选、复盘统计

### 4.2 可观测性

统一结构化日志字段：

1. `trace_id`
2. `stage`（asr/extract/match/save/timeline/review）
3. `stock_code`
4. `category`
5. `duration_ms`
6. `status`
7. `error_code`

### 4.3 发布节奏

1. 每周一个可回归版本
2. 每个 Sprint 末进行一次“性能 + 准确性”双验收
3. 严禁在 Sprint 中途引入大规模技术栈切换

---

## 五、后续扩展预留（不进入本阶段）

1. RAG：用于跨主题语义检索与总结
2. 图数据库：用于多跳关系分析与知识图谱
3. 多主题插件：从股票扩展到对谈知识库/认知盲点体系

---

## 六、Definition of Done

每个里程碑必须同时满足：

1. 功能可用（路径通、无阻塞）
2. 性能达标（SLO 达标）
3. 数据可追溯（日志与文档一致）
4. 用户体验稳定（主链路无明显卡顿）
